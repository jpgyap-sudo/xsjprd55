=== BEST DEPLOYMENT ARCHITECTURE FOR YOUR TRADING BOT ===

VERCEL FAILED: "No more than 12 Serverless Functions on Hobby plan"
This app is a long-running trading system, not a serverless frontend.

--- THE ANSWER: VPS-ONLY DEPLOYMENT ---

For this project, the production model is:

  [Telegram] <--webhook--> [Public domain / reverse proxy]
                              |
                         [DigitalOcean VPS]
                              |
          +-------------------+-------------------+
          |                   |                   |
     [API Server]      [PM2 Workers]        [Playwright]
     Port 3000          trading/data/        crawlers
                        research/ops
          |
      [Supabase] <--> [Dashboard static assets]

--- OPERATING MODEL ---

- Use PM2 as the production process manager.
- Use Tailscale/private SSH for administration whenever available.
- Use a public domain only for ingress/webhooks, not for operator SSH.
- Keep one authoritative deploy flow.
- Use consolidated health endpoints:
  - /api/vps-health
  - /api/worker-health
  - /api/pm2-status

--- PUBLIC INGRESS ---

Use a domain for webhook traffic:

  bot.yourdomain.com  A-record -> your public VPS IP

Then use TLS:

  certbot --nginx -d bot.yourdomain.com

Telegram webhook:

  https://bot.yourdomain.com/api/telegram

--- PRIVATE OPERATIONS ---

Use the private/Tailscale address for SSH and deployment automation:

  ssh root@100.86.182.7

--- RECOMMENDED VPS BASELINE ---

- Ubuntu 22.04+
- 2 vCPU / 2 GB RAM minimum
- swap only if live memory pressure shows it is needed
- PM2 log rotation enabled
- nginx or Caddy as reverse proxy
- regular backup verification
- worker heartbeat monitoring enabled

--- SKIP VERCEL ENTIRELY ---

The VPS handles:
- API endpoints
- background workers
- Telegram webhook
- dashboard assets

Vercel is unnecessary for this workload.
